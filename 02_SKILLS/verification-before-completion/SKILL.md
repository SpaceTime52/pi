# 완료 주장 전 검증하기

## 개요

검증 없이 작업이 끝났다고 주장하는 것은 효율이 아니라 부정직이다.

**핵심 원칙:** 주장을 하기 전에, 언제나 먼저 증거를 제시한다.

**이 규칙의 문구를 어기는 것은 곧 이 규칙의 정신을 어기는 것이다.**

## 철칙

```
새로운 검증 증거 없이 완료를 주장하지 않는다
```

이 메시지 안에서 검증 명령을 실제로 실행하지 않았다면, 통과했다고 주장할 수 없다.

## 게이트 함수

```
어떤 상태를 주장하거나 만족감을 표현하기 전에:

1. 확인: 이 주장을 입증하는 명령이 무엇인지 확인한다.
2. 실행: 그 명령 전체를 실행한다(새롭게, 빠짐없이).
3. 읽기: 전체 출력을 읽고, 종료 코드와 실패 개수를 확인한다.
4. 검증: 출력이 이 주장을 실제로 뒷받침하는가?
   - 아니라면: 증거와 함께 실제 상태를 말한다.
   - 맞다면: 증거와 함께 그 주장을 말한다.
5. 그 후에만: 그때서야 주장을 한다.

어느 단계든 건너뛰면 그것은 검증이 아니라 거짓말이다
```

## 흔한 실패 사례

| 주장 | 필요 조건 | 충분하지 않은 것 |
|-------|----------|----------------|
| 테스트가 통과함 | 테스트 명령 출력: 실패 0건 | 이전 실행 결과, "통과할 것 같음" |
| 린터가 깨끗함 | 린터 출력: 오류 0건 | 일부만 확인, 추정 |
| 빌드가 성공함 | 빌드 명령: exit 0 | 린터 통과, 로그가 좋아 보임 |
| 버그가 수정됨 | 원래 증상을 테스트: 통과 | 코드 변경, 수정됐다고 가정 |
| 회귀 테스트가 동작함 | 레드-그린 사이클 검증 완료 | 테스트가 한 번 통과함 |
| 서브에이전트 작업이 완료됨 | VCS diff에 변경 사항이 보임 | 서브에이전트가 "성공"이라고 보고함 |
| 요구사항을 충족함 | 항목별 체크리스트 검토 | 테스트가 통과함 |

## 위험 신호 - 멈춰라

- "~일 것 같다", "아마", "~처럼 보인다" 같은 표현을 쓰고 있음
- 검증 전에 만족감을 표현하려고 함("좋아요!", "완벽해요!", "끝났어요!" 등)
- 검증 없이 commit/push/PR 하려 함
- 서브에이전트의 성공 보고를 그대로 믿고 있음
- 부분 검증에 의존하고 있음
- "이번 한 번만"이라고 생각하고 있음
- 지쳤고 빨리 끝내고 싶어 함
- **검증을 돌리지 않았는데도 성공을 암시하는 어떤 표현이든 사용하려 함**

## 합리화 방지

| 핑계 | 현실 |
|--------|---------|
| "이제는 될 것 같음" | 검증을 실행하라 |
| "확신함" | 확신은 증거가 아니다 |
| "이번 한 번만" | 예외는 없다 |
| "린터는 통과했음" | 린터는 컴파일러가 아니다 |
| "서브에이전트가 성공했다고 했음" | 독립적으로 검증하라 |
| "너무 피곤함" | 피로는 변명이 아니다 |
| "부분 확인이면 충분함" | 부분 검증으로는 아무것도 증명되지 않는다 |
| "다른 표현이라 규칙이 적용되지 않음" | 문구보다 원칙이 우선이다 |

## 핵심 패턴

**테스트:**
```
좋음  [테스트 명령 실행] [확인: 34/34 통과] "모든 테스트가 통과했습니다"
나쁨 "이제 통과할 것 같습니다" / "맞아 보입니다"
```

**회귀 테스트(TDD 레드-그린):**
```
좋음  작성 -> 실행(통과) -> 수정 되돌리기 -> 실행(반드시 실패) -> 복원 -> 실행(통과)
나쁨 "회귀 테스트를 작성했습니다" (레드-그린 검증 없이)
```

**빌드:**
```
좋음  [빌드 실행] [확인: exit 0] "빌드가 통과했습니다"
나쁨 "린터는 통과했습니다" (린터는 컴파일을 확인하지 않는다)
```

**요구사항:**
```
좋음  계획 다시 읽기 -> 체크리스트 작성 -> 항목별 검증 -> 누락 또는 완료 상태 보고
나쁨 "테스트가 통과했으니 이 단계는 완료입니다"
```

**서브에이전트 위임:**
```
좋음  서브에이전트 성공 보고 -> VCS diff 확인 -> 변경 검증 -> 실제 상태 보고
나쁨 서브에이전트 보고를 그대로 믿음
```

## 왜 중요한가

24개의 실패 기억에서 드러난 것:
- 인간 협업자가 "당신 말을 못 믿겠다"고 했고, 신뢰가 깨졌다.
- 정의되지 않은 함수가 배포되어 실제로 크래시가 날 뻔했다.
- 요구사항이 빠진 채 배포되어 기능이 불완전했다.
- 잘못된 완료 판단 때문에 시간 낭비, 방향 전환, 재작업이 발생했다.
- "정직은 핵심 가치다. 거짓말하면 대체된다."는 원칙을 위반한다.

## 적용해야 하는 시점

**반드시 먼저 적용할 때:**
- 성공/완료를 뜻하는 어떤 표현이든 말하기 전
- 만족감을 드러내는 어떤 표현이든 하기 전
- 작업 상태에 대해 긍정적으로 말하기 전
- 커밋, PR 생성, 작업 완료 처리 전
- 다음 작업으로 넘어가기 전
- 서브에이전트에 위임하기 전

**이 규칙이 적용되는 대상:**
- 정확히 일치하는 표현
- 바꿔 말한 표현과 동의어
- 성공을 암시하는 뉘앙스
- 완료나 정확성을 시사하는 모든 커뮤니케이션

## 핵심 요지

**검증에는 지름길이 없다.**

명령을 실행하라. 출력을 읽어라. 그다음에야 결과를 주장하라.

이 원칙은 타협할 수 없다.
