# 스트레스 인터뷰

`$ARGUMENTS`에 대해 **`verifier` + `reviewer` + `challenger`를 병렬 호출**해 교차 검토한다.

## 목적

- 구현하거나 수정한 내용을 배포 전에 압박하듯 점검한다.
- 실행 증거, 코드 리뷰, 반론과 리스크를 동시에 수집한다.
- 한 에이전트의 편향을 줄이고, 겹치는 지적과 서로 다른 지적을 비교한다.

## 실행 규칙

1. 먼저 검토 대상을 1~2문장으로 다시 정의한다.
2. `subagent help`를 아직 확인하지 않았거나 현재 세션에서 인터페이스가 불분명하면 먼저 확인한다.
3. 아래 3개를 **병렬로** 실행한다.
   - `verifier`: 테스트, 타입체크, 빌드, 재현 가능한 검증에 집중
   - `reviewer`: 정확성, 회귀 위험, 유지보수성에 집중
   - `challenger`: 숨은 가정, 실패 시나리오, 의사결정의 취약점에 집중
4. 세 결과를 합쳐 아래 기준으로 정리한다.
   - 공통 지적: 둘 이상이 비슷하게 지적한 항목
   - 독립 지적: 한 에이전트만 찾았지만 타당한 항목
   - 상충 지적: 서로 결론이 다른 부분
5. 에이전트 결과는 **있는 그대로 요약**하고, 근거 없이 임의로 판정하지 않는다.

## 권장 호출 프롬프트

- `verifier`: "$ARGUMENTS 를 검증해줘. 가능하면 테스트, 타입체크, 빌드, 재현 가능한 증거를 수집해줘."
- `reviewer`: "$ARGUMENTS 를 코드 리뷰해줘. 정확성, 회귀 위험, 유지보수성 위주로 봐줘."
- `challenger`: "$ARGUMENTS 에 대해 숨은 가정, 실패 시나리오, 취약한 결정 지점을 최대 3개의 질문으로 압박 검토해줘."

## 종합 응답 형식

최종 응답은 아래 순서로 간단히 정리한다.

1. `전체 판정`
   - 준비됨 | 수정 필요 | 진행 불가
2. `공통 발견사항`
   - 공통 지적만 추린다
3. `검증 결과`
   - 핵심 검증 결과
4. `리뷰 결과`
   - 핵심 리뷰 결과
5. `도전 질문`
   - 핵심 질문과 리스크
6. `심각도 분류`
   - 🔴 반드시 수정: 진행 차단 이슈, 정확성 오류, 재현 가능한 버그
   - 🟡 수정 권장: 유지보수성, 명확성, 저위험 개선
   - ⚪ 수정 보류: 근거 부족, 의도된 설계, 대규모 변경 필요
7. `권장 다음 단계`
   - 수정이 필요하면 가장 먼저 할 일 1~3개

## 2단계 리뷰 모드

`$ARGUMENTS`에 `--2pass` 또는 "2단계 리뷰"가 포함되면 아래 순서를 따른다.

### 1단계: 명세 적합성

목적: 구현이 요구사항, 계획, 명세를 **정확히** 충족하는지 확인한다.

1. `verifier`에게: 명세 대비 구현이 일치하는지 검증해달라고 요청
2. `reviewer`에게: 요구사항 누락이나 초과 구현을 집중적으로 리뷰해달라고 요청

판정:

- **누락 구현**: 명세에는 있는데 구현에는 없는 것
- **초과 구현**: 명세에는 없는데 구현에는 있는 것 → YAGNI 위반
- 누락 구현이나 초과 구현이 있으면 수정 후 1단계를 다시 실행한다.

### 2단계: 코드 품질

**1단계를 통과한 뒤에만** 진행한다.

1. `reviewer`에게: 정확성, 회귀 위험, 유지보수성 관점의 리뷰를 요청
2. `challenger`에게: 숨은 가정과 실패 시나리오를 압박 검토해달라고 요청

판정:

- 치명적/중요 이슈 → 수정 후 2단계를 다시 실행
- 경미한 이슈 → 기록만 하고 통과

**주의: 1단계 전에 2단계를 시작하지 않는다.** 명세를 충족하지 못한 상태에서 코드 품질을 논하는 것은 의미가 없다.

## 주의

- 세 결과가 모두 오기 전에 성급하게 결론 내리지 않는다.
- `verifier`가 실행 증거를 모으지 못하면 그 사실을 분명히 적는다.
- `challenger`의 질문은 가설일 수 있으므로, 검증된 사실과 구분해 표시한다.
- 사용자가 단순 요약만 원하면 장황하게 다시 풀어쓰지 말고 핵심만 정리한다.
